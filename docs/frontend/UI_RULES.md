# UI_RULES.md

## 1. 视觉方向
- 整体风格必须简洁、克制、可信，不做花哨装饰型界面。
- 用户端与管理端共用同一套设计语言，但信息密度不同：
  - 用户端优先保证问答流畅与阅读舒适。
  - 管理端优先保证扫描效率与状态判断效率。
- 页面视觉应体现“大厂后台”气质：
  - 主背景保持轻灰与白底，不使用高饱和大面积色块。
  - 强调对齐、留白、层级和节奏，不依赖装饰图案取胜。
  - 每个页面只保留一个视觉中心，避免多个 Hero、多个高对比工具条同时争抢注意力。

## 2. 色彩与字体
- 推荐 CSS 变量：
```css
:root {
  --bg-base: #f3f5f7;
  --bg-panel: #ffffff;
  --bg-surface: #f8fafc;
  --text-primary: #0f172a;
  --text-secondary: #475569;
  --text-muted: #94a3b8;
  --line-soft: #e2e8f0;
  --brand-main: #2563eb;
  --brand-accent: #0f766e;
  --ok: #16a34a;
  --warn: #f59e0b;
  --danger: #dc2626;
}
```
- 中文字体优先 `"Source Han Sans SC"`, `"Noto Sans SC"`, `"PingFang SC"`, sans-serif。
- 英文与数字可使用 `"IBM Plex Sans"`, `"Segoe UI"`, sans-serif。
- 代码与 ID 字体可使用 `"JetBrains Mono"`, `"Cascadia Mono"`, monospace。

## 3. 间距与版式
- 使用 8pt 间距体系：`4 / 8 / 12 / 16 / 24 / 32`。
- 页面主体横向留白建议 `24px~32px`。
- 主内容区建议设置稳定的最大宽度，桌面端保持统一版心，不要任由内容无限拉伸。
- 卡片、按钮和输入控件圆角统一控制在 `8px` 左右，同一页面不混用过多圆角层级。

## 4. 组件外观
- 按钮：
  - 高频轻操作优先图标按钮。
  - 主操作允许保留文字，但在高密度管理端优先收敛为单个主图标按钮。
  - 危险操作必须使用显著危险色，并保留二次确认。
- 卡片：
  - 优先使用白底、轻边框、轻阴影。
  - 阴影用于层级区分，不用于制造悬浮炫技效果。
  - Hero 卡片允许使用极轻的顶部品牌渐变线，但正文区域必须克制。
- 表格：
  - 表头与正文对比度应克制，不做高饱和大面积色块。
  - 行 hover 可有轻微背景变化，但不能造成跳闪感。
  - 管理端表格推荐使用低对比浅底表头、细分隔线和轻 hover，不使用厚重边框或深色表头。
  - 紧凑模式与舒适模式必须统一由密度控件切换。
  - 管理端表格中的主信息列（如名称、摘要）必须优先保留可读宽度，元数据列应主动压缩或允许横向滚动，禁止把中文主字段挤成竖排断行。
- 表单：
  - 标签使用稳定中文，不展示内部字段名。
  - 校验提示必须是可读中文。
  - 输入控件圆角与高度应统一，避免不同页面风格漂移。
  - 管理端创建表单默认仅展示高频字段，低频参数应收纳进“高级设置”等折叠区，避免首屏信息过载。
  - 独立创建页不得只保留一根窄表单悬在大面积留白中；优先使用“主表单 + 辅助摘要栏/步骤提示”双栏结构，把空白转化为有组织的信息层级。
  - 管理端编辑弹窗优先使用轻说明 + 单栏参数布局，避免在窄容器内堆叠过多并列输入。

## 5. 门户壳层规则
- 左侧侧栏承担导航、身份、端口切换与登录退出，不再重复保留顶部功能条。
- 侧栏必须采用纵向撑满布局：
  - 导航区占据剩余高度。
  - 身份与端口切换卡片固定贴底。
  - 与右侧主工作区保持垂直基线一致。
- 管理端桌面侧栏必须优先保证导航区高度：
  - 品牌区允许展示产品识别，但应保持紧凑，不得挤压主导航可视区域。
  - 底部身份区与端口切换区应收敛为低占高卡片，避免与品牌区一起把导航区夹小。
- 桌面端管理壳层必须使用“右侧内容区独立滚动”的滚动策略：
  - 左侧侧栏固定占满视口高度。
  - 右侧内容超长时只允许主内容区滚动，不允许整页滚动导致侧栏留白。
- 图标按钮必须提供 `Tooltip` 或 `aria-label`。
- 品牌区、导航区、底部身份区应形成清晰三段式布局，边距与圆角统一。

## 6. 用户端规则
- `/app/ask` 使用聊天工作台布局：
  - 左侧为会话侧栏。
  - 右侧为消息区。
- 问答页必须保持主流 AI Chat 的单焦点体验：不展示会话 KPI、上下文大卡片或多组状态面板，避免干扰阅读和提问。
- 用户消息固定在右侧，助手消息固定在左侧。
- 输入区固定在消息卡片底部，只允许消息列表滚动。
- 证据面板通过弹窗承载，不保留常驻右栏。
- 用户端如需图表，只允许加入轻量摘要图：
  - 位置在消息区顶部。
  - 不得打断消息流与输入区主路径。
  - 优先展示会话结构与证据覆盖。
  - 优先使用紧凑摘要带或细条进度表达，不使用会遮挡消息区的大卡片图表。
- 聊天气泡需保持稳定宽度与阅读节奏：
  - 用户气泡与助手气泡的视觉差异应主要来自位置与轻微底色，不使用重边框和重色块。

## 7. 管理端规则
- 管理端页面顶部统一使用紧凑页头：
  - 页面标识
  - 页面标题
  - 信息说明图标
  - 3 到 4 个核心指标卡
  - 首屏应优先露出主表格或任务内容，页头与指标卡不得成为页面主体。
- 管理端工作台优先使用双栏结构：
  - 左侧表单/输入
  - 右侧列表/结果
- 筛选条统一使用左右两组结构：
  - 左侧放筛选条件与快捷开关
  - 右侧放计数与密度切换
- 图表用于增强判断，不替代表格与指标卡。
- 当页面只需要传达少量统计摘要时，优先使用页头指标卡或工具条内联状态胶囊，不额外拆出独立图表卡片。
- 当页面只需要传达轻量统计时，优先使用横向摘要带或轻量摘要胶囊，不要默认堆四张大统计卡。
- 当“创建/上传”和“列表/结果”属于两条独立任务流时，优先拆分为独立路由页面；管理页优先保证筛选、表格和状态面板的扫描效率。
- 拆分后的管理页仍需保留足够的容器留白；外层卡片与内层面板之间必须有缓冲边距，禁止工具条或表格直接贴近左边界。
- 知识库列表页与其他管理列表页使用同一套留白基线；即使未拆成新路由，也不得继续保留“零内边距嵌套面板”的贴边布局。
- 当单个管理工具条同时承担“主操作 + 搜索 + 筛选 + 密度切换”时，优先拆成两层；第一层放主操作与主检索，第二层放筛选、统计和视图控制。
- 页头已经承担关键统计时，工具条不得重复堆叠同类统计胶囊；优先保留筛选、刷新和密度等直接操作。
- 管理端的卡片、工具条、表格头部必须使用统一圆角、边框、阴影和控件高度。
- 桌面端管理页的工具条、表格行和指标卡默认使用紧凑比例；只有确实需要阅读长文本时才切回舒适密度。
- 后台表格列宽必须按优先级分配：主信息列固定并保留可读宽度，状态/数字/时间列主动压缩，操作列固定右侧；中屏和窄屏允许横向滚动，不允许把长中文名称挤成竖排。

## 8. 图表规则
- 不额外引入重量级图表依赖时，优先使用轻量原生图表组件。
- 优先使用三类图：
  - 结构分布图
  - 状态对比图
  - 轻量趋势图
- 图表必须服务于判断，不做装饰型可视化。
- 图表卡片需与页面其他卡片保持同一视觉系统：
  - 同一圆角
  - 同一边框强度
  - 同一标题层级

## 9. 响应式规则
- `>=1200px`：桌面双栏或多卡片布局。
- `768px~1199px`：允许折叠为单栏或上下堆叠。
- `<768px`：优先保障问答阅读、引用查看与表单可操作性。

## 10. 可访问性
- 所有图标按钮必须有 `aria-label` 或 `Tooltip`。
- 文字与背景必须满足基本可读对比度。
- 键盘操作必须可关闭弹窗、触发主要提交与聚焦关键按钮。

## 11. 正式业务界面收敛（2026-04）
- 页面背景统一使用轻灰底与白底内容面板，避免装饰性光斑、强渐变和大面积高饱和色块。
- 卡片、按钮、输入框、弹窗和表格容器统一使用低圆角，默认不超过 `8px`，只通过细边框和轻阴影表达层级。
- 管理端侧栏不得出现横向滚动条，品牌名称与导航文字必须保持可读，不允许被挤压成异常断行。
- 问答页保持聊天工作台主路径，输入区固定在底部，消息区和侧栏分别承担滚动，不用大块装饰卡片抢占首屏注意力。
- 错误提示主文案必须优先使用前端中文映射；非统一错误体的 HTTP 5xx 也应归一为“服务内部异常”类提示，原始错误只放在次级排障信息里。

## 12. 主流业务工作台风格（2026-04 第二轮）
- 侧栏导航使用“图标 + 功能名称 + 一行说明”的信息结构，帮助演示者快速说明模块价值，不只保留孤立菜单名。
- 品牌区使用紧凑标识、产品名、端口说明和当前视角标签，避免大面积品牌卡抢占工作区空间。
- 管理端页面以“任务页头 + 指标摘要 + 操作工作台”为主结构，视觉重点落在筛选、表格、任务状态和风险提示。
- 问答页空态必须承担业务引导，提供可直接填入输入框的示例问题；不能只显示“暂无数据”。
- 工具条与会话操作允许使用文字按钮，关键操作应让用户一眼识别，不只依赖图标含义。

## 13. 高级业务质感补充（2026-04 第三轮）
- 全局交互反馈使用细边框、轻底色、弱位移和统一焦点阴影，避免强动效、强发光和装饰性渐变。
- 问答页空态除示例问题外，只允许用轻量提示说明证据约束、拒答策略和引用定位，不能拆成仪表盘。
- 会话顶部只保留当前知识库和生成状态，不再展示会话统计 KPI。
- 输入区需要明确表达当前知识库上下文；已选知识库可用低饱和状态胶囊展示，避免用户误以为跨库查询。
- 示例问题优先放在空态，不在长期输入区下方常驻堆叠。
- 管理端表格 hover、侧栏菜单 hover、按钮焦点态必须使用同一套视觉节奏，形成稳定、可键盘操作的企业级体验。

## 14. 管理端与登录页质感补充（2026-04 第四轮）
- 管理端统一采用低噪声工作台布局：浅色固定侧栏、紧凑品牌区、清晰导航说明、右侧内容独立滚动。
- 管理端页头指标只表达关键判断，不得把筛选条件、状态胶囊和重复统计继续堆叠到工具栏。
- 管理端卡片、页头、工作台面板、表格容器使用同一套边框、圆角、阴影和 hover 节奏，避免不同页面像临时拼接。
- 表格优先保证中文主字段可读性，表头使用低饱和浅底，不使用高对比深色表头。
- 登录页采用产品级双栏入口：左侧表达产品能力与可信边界，右侧聚焦账号输入，不增加第二套登录路径。
- 登录页允许使用轻量网格背景和深色品牌面板，但表单区域必须保持白底、高对比和明确焦点态。

## 15. 管理端录入页补充（2026-04 第五轮）
- 创建、上传、评测集录入页必须使用“主表单 + 侧栏摘要/步骤”的结构，避免单根表单悬浮在大面积留白中。
- 表单底部主操作允许 sticky，但不得遮挡校验提示；移动端必须退化为普通纵向按钮组。
- 侧栏摘要只展示用户已填内容、当前策略和后续动作，不承载第二套提交入口。
- 登录页能力概览只用于解释产品边界，不得替代登录表单或增加额外认证流程。

## 16. 通用状态体验补充（2026-04 第六轮）
- `loading / empty / error` 状态必须使用统一工作台壳层，不得直接暴露默认组件造成 Demo 感。
- 空态需要解释“为什么没有内容”和“下一步如何出现内容”，不能只显示“暂无数据”。
- 错误提示必须把用户可执行下一步、后端返回、`request_id` 和错误码分层展示，便于演示和排障。
- 状态组件不得新增业务接口或第二套操作入口，只负责解释当前状态并保持页面节奏。

## 17. 浏览器质检补充（2026-04 启动检查轮）
- 管理端卡片的装饰性伪元素不得制造页面级横向滚动；外层工作区必须约束 `overflow-x`，表格内部横向滚动仍由表格容器承担。
- 登录页安全提示标签在桌面端必须保持完整短句，不得把 `request_id` 等排障关键词拆成异常断行；移动端可自然换行。
- 密码输入必须声明合适的 `autocomplete`，登录使用 `current-password`，创建或重置密码使用 `new-password`。
- 浏览器控制台不得保留可修复的组件库警告；路由、表单和加载态组件要优先使用项目统一壳层与已验证配置。
