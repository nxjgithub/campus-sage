# CampusSage（CSage）项目说明文档

## 1. 项目简介

**CampusSage（简称 CSage）** 是一个面向高校场景的“可追溯”知识库智能问答系统。它把分散在教务处通知、办事指南、规章制度、奖助学金说明等文档中的信息统一入库，通过 **向量检索（Vector Search）+ 大模型生成（vLLM）** 的方式，为师生提供“问一句就能得到答案，并且能看到证据出处”的服务。

该项目采用 **RAG（Retrieval-Augmented Generation，检索增强生成）** 架构：模型回答时必须基于检索到的证据片段（chunks），并输出引用来源（文档名、页码/章节、命中片段），在证据不足时拒答或提示进一步查询路径，从而减少“看似合理但实际编造”的风险。

---

## 2. 背景与痛点

校园知识的典型问题是：信息“存在”，但难“获取”和难“统一”。

* **信息碎片化**：制度与流程散落在多个网站、PDF、群通知、下载中心；
* **重复咨询**：同类问题（选课、补考、奖学金材料、宿舍报修等）高频出现，占用老师/行政时间；
* **版本混乱**：制度更新后旧截图、旧PDF仍在传播，导致误解与扯皮；
* **可信度问题**：普通聊天机器人容易“编”，无法作为流程依据。

CampusSage 的价值在于：**把“找信息”变成“问问题 + 看证据”**，让答案可复核、可追责、可维护。

---

## 3. 项目目标与范围

### 3.1 项目目标（必须达成）

1. 支持将校园相关文档上传并入库（解析、切分、向量化、写入向量库）；
2. 支持基于指定知识库进行问答（检索 → 生成）；
3. 输出**答案 + 引用证据**（至少包含文档名、页码/章节、命中原文片段）；
4. 证据不足时触发**拒答策略**（不胡编）；
5. 记录问答日志与会话历史，便于复现与评测；
6. 提供基础后台能力：文档管理、重建索引、配置检索参数。

### 3.2 项目范围（建议先聚焦）

建议优先选择 **“可验证、规则明确”** 的校园知识作为初期知识库，例如：

* 教务规则：选课/补考/缓考/学分认定/毕业要求；
* 奖助学金：条件、材料清单、时间节点、评定流程；
* 宿舍与后勤：报修、迁宿、门禁规定；
* 图书馆：借阅规则、超期处理。

### 3.3 非目标（避免早期失控）

* 不追求全校百科式覆盖；
* 不优先做复杂前端（初期可用 Swagger 或极简页面演示）；
* 不强制做模型微调（先在闭环稳定后再做实验对比）。

## 4. 技术方案与选型

### 4.1 技术栈

* 后端框架：**FastAPI**
* 向量数据库：**Qdrant**（或 pgvector）
* 推理服务：**vLLM**（LLM 推理加速，建议 OpenAI-compatible 接口方式接入）
* Embedding 模型：建议使用中文表现较稳的 bge 系列（如 `bge-m3` / `bge-large-zh`）
* 可选重排：`bge-reranker`（提升检索相关性）
* 数据库：SQLite（起步）/ MySQL（后期）
* 文档解析：PDF 文本解析为主（后续可扩展 docx/html）
* 开发方式：**Codex + PyCharm**（小步迭代 + 测试驱动）

### 4.2 核心原则（工程与论文都好写）

* **证据优先**：不允许无证据生成；
* **引用可定位**：每条引用必须能回指到文档位置；
* **模块边界清晰**：ingest / retrieve / generate / eval 分层；
* **可评测**：必须能跑离线评测集，产出检索指标与延迟数据。

---

## 5. 系统架构概览

系统可拆为四个核心模块：

1. **Ingest（入库）**
   上传文档 → 解析文本 → 切分 chunk → embedding → 写入向量库（带 metadata）
   并记录文档与入库任务状态（成功/失败/原因）。

2. **Retrieve（检索）**
   对用户问题生成 query embedding → TopK 向量检索 →（可选 rerank）→ 返回候选证据 chunks。

3. **Generate（生成）**
   将证据 chunks 组装为上下文 → 调用 vLLM 生成答案 → 输出引用列表 → 若证据不足触发拒答。

4. **Eval（评测）**
   维护小型评测集（问题 + 标准证据）→ 输出 Recall@K / MRR 等检索指标 + 延迟统计。

---

## 6. 数据流（两条生命线）

### 6.1 入库流程（Ingest Flow）

1. 管理员上传文档（PDF）；
2. 服务解析提取文本，并保留定位信息（页码/章节）；
3. 文本切分为 chunks（chunk_size + overlap）；
4. 每个 chunk 生成 embedding；
5. 将 embedding 写入向量库，并附带 metadata（doc_id、page、section、chunk_id…）；
6. 文档状态更新为 indexed 或 failed（记录错误原因）。

### 6.2 问答流程（RAG Flow）

1. 用户选择知识库并提问；
2. 生成 query embedding；
3. 在指定知识库中向量检索 TopK chunks；
   4.（可选）重排，提高相关性；
4. 拼接上下文并调用 vLLM；
5. 返回：答案 + 引用证据 +（可选）耗时/检索分数；
6. 写入会话与日志，支持回放与评测分析。

## 7. 功能清单（成品级目标）

### 7.1 用户端（演示重点）

* 选择知识库 → 提问 → 获取答案；
* 引用证据列表（文档名 + 页码/章节 + snippet）；
* 会话历史；

### 7.2 管理端（工程加分）

* 知识库管理（创建/配置 TopK、chunk、阈值、prompt）；
* 文档管理（上传、删除、重建索引、查看入库状态）；
* 日志查看（问答耗时、失败原因、命中证据）；
* 评测管理（评测集、跑指标、导出报告）。

## 8. 关键设计契约（必须提前定死，避免返工）

### 8.1 Chunk Metadata（写入向量库的字段）

每个 chunk 至少包含：

* kb_id、doc_id、doc_name
* page_start/page_end（或 null）
* section（可选）
* chunk_id、chunk_index
* text（原文片段）

> 这套 metadata 决定了你引用是否能“定位”，是成品可信度的根基。

### 8.2 引用输出格式（API Response）

问答接口必须返回：

* `answer: str`
* `citations: List[{doc_id, doc_name, page_start, page_end, section, snippet}]`
* `refusal: bool`（证据不足时为 true，并附提示语）

## 9. 计划与里程碑（建议按周推进）

### 里程碑 M1：MVP 闭环（最重要）

* Qdrant + FastAPI 跑起来；
* 上传 PDF → 入库成功；
* 提问 → 检索 → vLLM 生成；
* 返回答案 + 引用；
* 有日志与会话记录。

### 里程碑 M2：质量提升

* 元数据过滤（按知识库/文档标签/时间）；
* 引入 rerank；
* 拒答策略完善（阈值 + 覆盖度）；
* 引用展示更清晰（snippet 扩展、编号对应）。

### 里程碑 M3：评测与论文材料

* 小评测集（20~50条）；
* 输出 Recall@K、MRR、延迟（平均/P95）；
* 做参数对比实验（chunk_size、TopK、是否 rerank）。

### 里程碑 M4：可演示与答辩

* 最简前端或后台页面；
* 演示脚本（3个知识库 + 10个问题）；
* 部署说明与一键启动（docker compose）。

---

## 10. 评测与验收标准（答辩可直接用）

验收标准建议固定为：

1. 能成功入库 ≥ 10 份校园文档；
2. 对于评测集问题，Recall@5 达到可用水平（以你数据为准，给出实测值）；
3. 每次回答都能输出 ≥ 2 条引用（或根据问题性质合理变化）；
4. 证据不足时拒答率合理（不会硬编）；
5. 输出响应耗时统计（平均、P95），并给出 vLLM 加速说明。

## 11. 使用 Codex + PyCharm 的开发方式（推荐流程）

* 每次让 Codex 只完成一个小任务（一个 endpoint / 一个模块）；
* 每次任务都有“Definition of Done”：

    * 必须加测试（pytest）；
    * 必须过 lint（ruff）；
    * 不得修改无关文件；
* 仓库中提供指导文件：

    * `AGENTS.md`（总规则）
    * `docs/RAG_CONTRACT.md`（引用与 metadata 契约）
    * `docs/API_SPEC.md`（接口形状）
    * `.agents/skills/`（可选：沉淀常用技能）
